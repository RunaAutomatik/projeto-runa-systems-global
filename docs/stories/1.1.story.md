# Story 1.1 — FFmpeg compositor for slide-based reels

## Story Statement

As FREYJA (content operator), I want to send a brief with `type: reel` and receive
a rendered MP4 9:16 video file, so that I can publish Reels to @arthsystems_
without any manual video editing.

## Background

`apps/content-worker/src/video/compositor.js` currently returns `status: 'pending'`
for all reel/video requests. This is a documented stub waiting for Stage 1 implementation.

The architecture decision (content-automation-architecture.md) chose **FFmpeg** over
Remotion for Stage 1 because: 1-2 days effort vs 4-5 days, covers 80% of use cases,
no AWS Lambda dependency, uses existing Puppeteer PNG slides as input.

## Acceptance Criteria

- [ ] **AC1:** `POST /render` with `type: reel` returns `{ content_id, type, videoUrl, caption }` (not `status: pending`)
- [ ] **AC2:** Output is a valid MP4 file, 9:16 aspect ratio (1080×1920), minimum 3 seconds
- [ ] **AC3:** Video is uploaded to Supabase Storage and a public URL is returned
- [ ] **AC4:** Given N slides (PNG from carousel renderer), FFmpeg concatenates them into the MP4
- [ ] **AC5:** Each slide displays for a configurable duration (default: 3 seconds per slide)
- [ ] **AC6:** `brief.type === 'video'` follows the same path as `brief.type === 'reel'`
- [ ] **AC7:** If FFmpeg binary is not available, returns a clear error (no crash, no silent fail)

### FFmpeg Command Reference (Stage 1)

```bash
# Concat N slides into MP4 9:16, each slide shown for DURATION seconds
ffmpeg -framerate 1/DURATION \
  -i /tmp/content-{id}/slide-%03d.png \
  -vf "scale=1080:1920:force_original_aspect_ratio=decrease,pad=1080:1920:(ow-iw)/2:(oh-ih)/2" \
  -c:v libx264 -pix_fmt yuv420p \
  /tmp/content-{id}/video.mp4
```

Use `child_process.execFile('ffmpeg', [...args])` — NOT `exec` (avoids shell injection).
